# QEMU 用户态网络（SLIRP）让虚机联网 —— 调试记录

> 目标：让 `/home/wz/tests/qemu_debug_qcow2/run_qemu_qcow2_sandbox.sh` 启动的 QEMU 虚机能访问外网。
> 结论：最终改用 QEMU 用户态网络（SLIRP），并解决了「QEMU 未编入 slirp」和「嵌套 slirp 网段撞车」两个叠加问题，虚机成功联网并做了开机持久化。

---

## 1. 环境背景

| 项 | 值 |
|----|----|
| 架构 / 系统 | aarch64 / openEuler 24.03 LTS-SP2 |
| QEMU | 自编译 `~/qemu/build/qemu-system-aarch64`（10.2.90 / v11.0.0-rc0） |
| 加速 | TCG（**无 KVM**，本机 `/dev/kvm` 不可用） |
| 特殊点 | **本机的「host」本身也是一台用 slirp 起的虚机**（外层网段 `10.0.2.0/24`）——这是后面撞车的根因 |
| 虚机镜像 | `openEuler-24.03-LTS-SP2-aarch64_src.qcow2` |
| 虚机登录 | `root` / `wangroot` |

原脚本的网络用的是 `-netdev tap` 接入内部网桥 `br0`（`192.168.100.0/24`），该网桥孤立、未配 NAT，所以虚机**出不了网**。这套 tap+bridge 拓扑本是为 src↔dst **热迁移**设计的。

---

## 2. 方案选择

两条路：

- **方案 A：bridge + tap + NAT** —— 保留 `br0` 热迁移拓扑，在宿主机加 IP 转发 + iptables MASQUERADE。能上外网且保留 src↔dst 互通，但需 root。
- **方案 B：QEMU 用户态网络（`-netdev user` / SLIRP）** —— QEMU 自带 NAT，开箱即上网、无需 root/bridge；缺点是虚机拿 `10.0.2.x`，与 `br0` 热迁移拓扑不互通。

**最终选择方案 B**（sandbox 场景只需上外网，不需热迁移互通；热迁移仍用 `run_qemu_qcow2_src.sh`）。

---

## 3. 踩坑与解决（关键部分）

「改了还是不通」的根本原因是**两个不同的问题先后叠加**。

### 坑 1：QEMU 编译时没有 slirp

把脚本改成 `-netdev user` 后，一启动就报：

```
qemu-system-aarch64: -netdev user,id=net0: network backend 'user' is not compiled into this binary
```

排查：

```bash
# meson 日志确认 slirp 未启用
grep -i slirp ~/qemu/build/meson-logs/meson-log.txt
#  → Run-time dependency slirp found: NO (tried pkgconfig)

# 系统只装了运行库，缺 devel（头文件/pkg-config）
rpm -q libslirp libslirp-devel
#  → libslirp-4.7.0 已装；libslirp-devel 未安装

# 而且本机没有配置任何 dnf 仓库，装不了 devel 包
sudo dnf repolist        # 输出为空
```

**解决：用 QEMU 自带的 slirp 子项目重新编译**（绕过系统 devel 包）。
QEMU 源码树自带 `subprojects/slirp.wrap`，meson 在 `--enable-slirp` 且系统无 slirp 时会自动从 git 拉取并内建编译。

```bash
# 前提：能访问 gitlab.freedesktop.org（已确认 git ls-remote 通）
cd ~/qemu/build
../configure --target-list=aarch64-softmmu --enable-slirp
#  → 配置输出中 Subprojects: slirp : YES

ninja qemu-system-aarch64
#  → Linking target qemu-system-aarch64，exit 0
```

验证：

```bash
~/qemu/build/qemu-system-aarch64 -machine none -netdev help | grep -w user
#  → user   （不再报 not compiled）
```

> 注意：slirp 作为子项目在 `build/` 内部编译，源码**不会**落到 `~/qemu/subprojects/libslirp*`，所以用「subprojects 下有没有 libslirp 目录」判断是否成功是错的，以 `-netdev help` 列出 `user` 为准。

### 坑 2：嵌套 slirp 网段撞车（本次最隐蔽的问题）

重编译后 `-netdev user` 能用了，虚机拿到 `10.0.2.15`、能 ping 通网关 `10.0.2.2`，**但 ping 不通外网 `223.5.5.5`**。同时观察到一个奇怪现象：**「host」和 guest 的 IP 都是 `10.0.2.15`**。

根因：**本机的「host」本身是用 slirp 起的虚机**，外层已占用 `10.0.2.0/24`；而内层 sandbox 的 slirp **默认也是 `10.0.2.0/24`**，两层完全重叠：

```
物理机
 └ QEMU-slirp（外层） → host 虚机    10.0.2.15，网关 10.0.2.2
      └ QEMU-slirp（内层） → sandbox 虚机 10.0.2.15，网关 10.0.2.2   ← 网段完全重叠
```

两层 NAT 用同一套地址空间，内层出站到「网关 10.0.2.2」后绕不出去 → **网关能通、公网不通**。

**解决：把内层 slirp 改到不同网段 `10.0.3.0/24`。** 修改 `run_qemu_qcow2_sandbox.sh`：

```
-netdev user,id=net0,net=10.0.3.0/24,host=10.0.3.2,dns=10.0.3.3 \
-device virtio-net-pci,netdev=net0,mac=52:54:00:12:34:01 \
```

改完后虚机 DHCP 拿到 `10.0.3.15`，`curl www.baidu.com` 返回 **HTTP 200**。

### 坑 3：用 ping 测外网不可靠（认知陷阱）

**SLIRP 对 ICMP（ping）访问公网的支持有限**，经常出现「ping 外网 IP 不通，但 TCP 完全正常」。调试过程中一度被 ping 不通误导。

**正确做法：用 `curl` / TCP 验证连通性**，而不是 ping：

```bash
curl -sS -m 12 -o /dev/null -w 'HTTP=%{http_code}\n' http://223.5.5.5/       # 测 NAT 出口
curl -sS -m 12 -o /dev/null -w 'HTTP=%{http_code} ip=%{remote_ip}\n' http://www.baidu.com/   # 测 DNS+外网
```

## 4. 开机持久化（免每次手动配网）

问题：虚机开机后 `enp0s1` 是 DOWN、无 IP、无路由，需每次手动 `dhclient`。
探测发现：**NetworkManager 与 systemd-networkd 都未启用**（`systemd-networkd` 甚至 not-found），所以没人管这块网卡。

解决：**启用 NetworkManager + 建自动 DHCP 连接**（在虚机内执行一次，写入 qcow2 持久保存）：

```bash
systemctl enable --now NetworkManager
nmcli con add type ethernet ifname enp0s1 con-name enp0s1 \
      ipv4.method auto connection.autoconnect yes
nmcli con up enp0s1
```

NetworkManager 会自动：开机拉起 `enp0s1` 走 DHCP、把 DNS 写入 `/etc/resolv.conf`（`# Generated by NetworkManager` + `nameserver 10.0.3.3`）。

**验证（重启后不敲任何命令）**：`enp0s1` 自动得到 `10.0.3.15`、DNS 自动就绪、`curl www.baidu.com` 返回 200。登录横幅直接显示 `IP address: 10.0.3.15`。

## 5. 最终配置与用法

### 脚本网络参数（`run_qemu_qcow2_sandbox.sh`）

```
-netdev user,id=net0,net=10.0.3.0/24,host=10.0.3.2,dns=10.0.3.3 \
-device virtio-net-pci,netdev=net0,mac=52:54:00:12:34:01 \
```

### 网段规划

| 地址 | 角色 |
|------|------|
| `10.0.3.15` | sandbox 虚机（slirp 分配的客户机地址） |
| `10.0.3.2`  | 虚拟网关（QEMU 进程本身，负责 NAT） |
| `10.0.3.3`  | 虚拟 DNS |
| `10.0.2.0/24` | **外层 host 虚机占用**，内层务必避开 |

### 日常用法

```bash
cd /home/wz/tests/qemu_debug_qcow2
./run_qemu_qcow2_sandbox.sh
# 登录 root / wangroot —— 开机即联网，可直接 curl / yum / git
```

---

## 6. 结论速查

1. **`-netdev user` 报 `not compiled`** → QEMU 缺 slirp，用 `--enable-slirp` 重编译（走自带 `slirp.wrap` 子项目，不依赖系统 devel 包）。
2. **能通网关、不通公网，且内外 IP 相同** → 嵌套 slirp 网段撞车，用 `net=/host=/dns=` 给内层换网段（本例 `10.0.3.0/24`）。
3. **测外网别用 ping** → SLIRP 对 ICMP 外网支持有限，用 `curl`/TCP。
4. **每次开机要手动配网** → 启用 NetworkManager 建 autoconnect DHCP 连接，持久化到镜像。
5. 用户态网络与 `br0` 热迁移拓扑**不互通**；热迁移场景用 `run_qemu_qcow2_src.sh`。

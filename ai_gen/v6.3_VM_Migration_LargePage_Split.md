# v6.3 虚拟机迁移时大页拆分流程分析

本文档分析 v6.3 版本（无 eager page splitting）在虚拟机迁移时触发大页拆分的完整流程。

## 背景

v6.3 版本**没有** `kvm_pgtable_stage2_split()` 函数，大页拆分是在 **Page Fault 时按需拆分**（On-demand split）。

---

## v6.3 vs 主线版本对比

| 特性 | v6.3（旧版） | 主线版本 |
|------|------------|---------|
| **拆分时机** | Page fault 时按需拆分 | 可提前（eager split）或按需 |
| **Split 函数** | 无 | `kvm_pgtable_stage2_split()` |
| **Dirty logging** | 触发 fault 时才拆分 | 可以提前全量拆分 |
| **迁移效率** | 可能传输多余数据 | 精确追踪 dirty page |
| **代码位置** | `user_mem_abort()` | `kvm_mmu_split_huge_pages()` |

## 详细流程

### 阶段1：启用 Dirty Logging

```bash
用户空间 (QEMU)
    │
    ▼
ioctl(KVM_SET_USER_MEMORY_REGION)  # 创建 memory slot
    │
    ├─► kvm_create_memslot()        # 分配脏页 bitmap
    │
    └─► 默认行为：不做任何拆分！
        （v6.3 没有 eager splitting）
```

### 阶段2：Guest 运行期间

```
┌────────────────────────────────────────────────────────────────┐
│                        Guest 运行                              │
│                                                                │
│   Guest 访问内存 ──► Stage-2 页表 ──► 物理内存                │
│                         │                                     │
│                         │ 64K Block Mapping                   │
│                         ▼                                     │
│                    物理内存 (64K 大页)                         │
│                                                                │
│   Guest 写入这块内存 ──► Stage-2 PTE 变 dirty                 │
│                                                                │
│   但是：KVM 只能追踪到"64K 大页"脏了，                         │
│         无法知道具体哪个 4K page 脏了                          │
└────────────────────────────────────────────────────────────────┘
```

### 阶段3：迁移时读取 Dirty Log

```
用户空间 (QEMU)
    │
    ▼
ioctl(KVM_GET_DIRTY_LOG, &log)    # 读取脏页
    │
    ▼
┌────────────────────────────────────────────────────────────────┐
│ kvm_vm_ioctl_get_dirty_log()    // virt/kvm/kvm_main.c        │
│                                                                │
│ 1. 遍历 memslot 的 dirty bitmap                                 │
│ 2. 找出哪些 GFN 被标记为 dirty                                  │
│ 3. 返回给用户空间                                              │
└────────────────────────────────────────────────────────────────┘
    │
    ▼
┌────────────────────────────────────────────────────────────────┐
│ 问题：dirty bitmap 是基于 GFN（Guest Frame Number）            │
│       每个 GFN = 4K page                                       │
│                                                                │
│ 但如果 stage-2 用的是 64K block mapping...                     │
│ 那 dirty bitmap 怎么标记？                                     │
└────────────────────────────────────────────────────────────────┘
```

---

## v6.3 的解决方案：Write Protect + Fault 触发拆分

**v6.3 不在 `KVM_GET_DIRTY_LOG` 时拆分，而是在 write protect 时触发 fault：**

```
┌────────────────────────────────────────────────────────────────┐
│ 用户空间读取 dirty log 后                                      │
│                                                                │
│ QEMU 调用: kvm_mmu_write_protect_pt_masked()                   │
│           或者 kvm_arch_mmu_enable_log_dirty_pt_masked()       │
│                                                                │
│ └─► 调用 kvm_pgtable_stage2_wrprotect()                         │
│       └─► 遍历地址范围，WRPROT 所有 leaf PTE                   │
│                                                                │
│ 对于 64K block PTE ──► 直接 write protect 这个 block           │
│                                                                │
│ 问题：整个 64K 都被保护了，只能知道"64K 脏了"                  │
└────────────────────────────────────────────────────────────────┘
    │
    ▼
┌────────────────────────────────────────────────────────────────┐
│ Guest 再次访问被保护的页                                        │
│                                                                │
│ Access ──► Stage-2 Fault ──► KVM 页错误处理                     │
│                    │                                           │
│                    ▼                                           │
│            user_mem_abort()            // mmu.c                 │
│                    │                                           │
│                    ▼                                           │
│        ┌───────────────────────┐                               │
│        │ 检查 force_pte 标志   │                               │
│        │ (logging_active)     │                               │
│        └───────────────────────┘                               │
│                    │                                           │
│                    ▼                                           │
│        kvm_pgtable_stage2_map()                                │
│                    │                                           │
│                    ▼                                           │
│     ┌──────────────────────────────┐                           │
│     │ stage2_map_walk_leaf()      │                           │
│     │                              │                           │
│     │ 当前 PTE: 64K block         │                           │
│     │ 需要映射: 4K page           │                           │
│     │                              │                           │
│     │ 执行:                        │                           │
│     │ 1. break block PTE         │                           │
│     │ 2. allocate 4K page table   │                           │
│     │ 3. map 16x4K PTEs          │                           │
│     └──────────────────────────────┘                           │
│                    │                                           │
│                    ▼                                           │
│        64K block ──► 16 x 4K pages 拆分完成！                   │
└────────────────────────────────────────────────────────────────┘
```

## 关键代码路径（v6.3）

### 1. user_mem_abort 中的 force_pte 设置

```c
// v6.3: arch/arm64/kvm/mmu.c:1215-1300
static int user_mem_abort(...) {
    bool logging_active = memslot_is_logging(memslot);
    
    // 如果启用了 dirty logging，强制使用 PTE
    if (logging_active) {
        force_pte = true;
        vma_shift = PAGE_SHIFT;  // 4K
    }
    
    // ...
    
    // 如果是 hugetlbfs 大页
    if (is_vm_hugetlb_page(vma) && !(vma->vm_flags & VM_PFNMAP)) {
        vma_shift = get_vma_page_shift(vma, hva);
        // 处理 CONT_PMD 等情况...
    }
    
    // ...
    
    // 映射时
    ret = kvm_pgtable_stage2_map(pgt, fault_ipa, vma_pagesize,  // vma_pagesize 可能是 4K
                     __pfn_to_phys(pfn), prot, memcache, ...);
}
```

### 2. Permission Fault 时的处理

```c
// v6.3: arch/arm64/kvm/mmu.c:1408-1415
/*
 * Under the premise of getting a FSC_PERM fault, we just need to relax
 * permissions only if vma_pagesize equals fault_granule. Otherwise,
 * kvm_pgtable_stage2_map() should be called to change block size.
 */
if (fault_status == ESR_ELx_FSC_PERM && vma_pagesize == fault_granule)
    ret = kvm_pgtable_stage2_relax_perms(pgt, fault_ipa, prot);
else
    ret = kvm_pgtable_stage2_map(pgt, fault_ipa, vma_pagesize,
                     __pfn_to_phys(pfn), prot, memcache,
                     KVM_PGTABLE_WALK_HANDLE_FAULT |
                     KVM_PGTABLE_WALK_SHARED);
```

## v6.3 的问题

1. **迁移效率低**：首次读取 dirty log 时，如果脏 block 包含大量干净的小 page，还是会传输整个 block

2. **Fault 风暴**：Guest 访问被保护的脏页时，会产生大量 Permission Fault

3. **拆分延迟**：只有在实际访问时才拆分，无法预先准备好

---

## 主线版本的改进

```
主线版本新增了 eager page splitting：
    │
    ▼
kvm_mmu_split_huge_pages(kvm, start, end)  // 可在迁移前调用
    │
    ├─► kvm_pgtable_stage2_split()
    │     └─► stage2_split_walker()
    │           └─► 预拆分所有大页为 4K pages
    │
    ▼
Stage-2 直接就是 4K page mapping
    │
    ▼
迁移时 dirty bitmap 精确到 4K page，无需 fault 触发
```

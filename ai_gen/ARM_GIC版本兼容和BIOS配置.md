ARM GIC版本兼容性以及BIOS GIC版本配置
======================================

-v0.1 2026.05.23 Sherlock init

简介：梳理ARM GIC各版本(GICv2/v3.x/v4.0/4.1/4.2/v5)之间的兼容性结论。分析从软件
      兼容性角度考虑，BIOS GIC版本的配置。用于开发人员对齐认知。


## ARM GIC兼容性

| 关系 | 结论 | 说明 |
|------|------|------|
| GICv3.0→3.1→3.2→3.3 | 兼容 | 增量超集, 新特性可发现且可选, 旧软件可跑新硬件 |
| GICv4.x对GICv3.x | 兼容 | GICv4是GICv3超集, 只用v3功能的软件在v4上正常 |
| GICv3对GICv2 | 有条件 | 可选legacy模式(FEAT_GICv3_LEGACY), 已deprecated |
| GICv4.0对GICv4.1 | 不兼容 | 直接注入编程模型断层, hypervisor须区分对待 |
| GICv4.1→4.2 | 兼容 | v4.2是v4.1超集(追加NMI属性) |
| GICv4.2蕴含v3.3 NMI | 是 | v4.2 = v4.1 + v3.3 NMI, 必然成立 |
| GICv4.1蕴含v3.3 NMI | 否 | 两者正交, v4.1本身不含NMI |
| GICv3.3蕴含v4.2 | 否 | 有物理NMI不代表有虚拟NMI注入 |
| GICv5对v3/v4(host) | 不兼容 | clean break, host需原生v5驱动 |
| GICv5 host跑v3 guest | 支持 | 靠FEAT_GCIE_LEGACY + 虚拟CPU接口 |

ARM GIC协议自身的兼容性一塌糊涂。我们重点看不兼容的点：

1. GICv4.0和GICv4.1不兼容
2. GICv4.0和GICv4.2不兼容
3. 对于NMI特性，GICv4.0/GICv4.1和GICv3不兼容
4. GICv5和GICv3/GICv4都不兼容

## 软件兼容性

当我们说软件兼容性的时候，我们在说什么？看一个例子，用户的老版本内核只支持了GICv4.0，
用户基于老版本内核做了无数开发，在一定时间内不希望升级这个内核版本，稳定性考虑，
老内核版本一般只做bugfix，不做大特性回合，比如，这个老内核不会回合GICv4.1的代码。

我们硬件升级了，支持了GICv4.1，因为GICv4.1不兼容GICv4.0，如上用户的老系统要正常
跑起来，就需要硬件保留GIC4.0的能力。

注意，只要保留老硬件的能力，并且可以呈现给用户，兼容性就没有问题。所以，BIOS的
默认配置不重要，重要的是有这个配置可以选。

## BIOS GIC版本配置

用户的软件版本支持的GIC特性可以固定在如上任意一个点，所以，理论上所有硬件不兼容
的点都要给出可选配置。所以，我们给出的BIOS选项应该包括：

1. GICv3.3(支持NMI)
2. GICv4.0 
3. GICv4.1
4. GICv4.2
5. GICv5

但是，GICv3.3的NMI是从GICv4.2回合的，用户应该不会基于GICv3使用NMI(需要在老版本内
核上回合NMI支持补丁)。所以，我们只要保留2-5的BIOS配置项就好。
